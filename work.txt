1. Project idea
---------------

FraudShield is a machine learning and MLOps project for detecting credit card fraud.

The project uses the IEEE-CIS Fraud Detection dataset from Kaggle. The final goal is to build a full production-style pipeline:

- load and validate fraud data
- create useful features
- train an XGBoost fraud detection model
- track model experiments with MLflow
- explain predictions with SHAP
- serve predictions with FastAPI
- store logs in PostgreSQL
- simulate data drift
- detect drift
- retrain the model when needed
- show metrics in a React dashboard and Grafana
- run checks with GitHub Actions

4. Feature mapping plan
-----------------------

These are the planned mappings from raw IEEE-CIS columns to easier project features.

Transaction amount:

    amount = TransactionAmt

Transaction time:

    TransactionDT is a relative time value in seconds.

    day_number = floor(TransactionDT / 86400)
    hour = floor((TransactionDT % 86400) / 3600)
    day_of_week = day_number % 7
    is_weekend = day_of_week in [5, 6]

Card features:

    card_network = card4
    card_type = card6

User proxy:

    user_key = combination of card and address columns
    user_key = card1 + card2 + card3 + card5 + addr1

This is not a real user ID. It is an approximate identity created from stable card/address signals.

Merchant/category proxy:

    merchant_key = combination of product and transaction pattern columns
    merchant_key = ProductCD + card4 + P_emaildomain

This is not a real merchant ID. It is a useful proxy because the dataset does not expose actual merchant IDs.


5. Planned feature formulas
---------------------------

Fraud rate:

    fraud_rate = fraud_count / total_count
    fraud_rate = mean(isFraud)

User transaction count over 30 days:

    user_tx_count_30d =
        count of previous transactions for the same user_key
        where transaction_time is within the last 30 days

User average amount over 30 days:

    user_avg_amount_30d =
        average TransactionAmt for the same user_key
        where transaction_time is within the last 30 days

User total amount over 30 days:

    user_total_amount_30d =
        sum of TransactionAmt for the same user_key
        where transaction_time is within the last 30 days

Merchant transaction count over 30 days:

    merchant_tx_count_30d =
        count of previous transactions for the same merchant_key
        where transaction_time is within the last 30 days

Merchant unique users over 30 days:

    merchant_unique_users_30d =
        number of unique user_key values for the same merchant_key
        where transaction_time is within the last 30 days

Velocity feature:

    velocity_1hr =
        count of previous transactions for the same user_key
        where transaction_time is within the last 1 hour

Important rule:

    Rolling features must be point-in-time safe.
    This means a row can only use transactions that happened before it.
    It must not use future transactions.


6. Model and metric formulas
----------------------------

Confusion matrix terms:

    TP = true positives
    TN = true negatives
    FP = false positives
    FN = false negatives

Precision:

    precision = TP / (TP + FP)

Meaning:

    Out of all transactions predicted as fraud, how many were actually fraud?

Recall:

    recall = TP / (TP + FN)

Meaning:

    Out of all real fraud transactions, how many did the model catch?

F1 score:

    F1 = 2 * precision * recall / (precision + recall)

Meaning:

    A balanced score between precision and recall.

ROC-AUC:

    Measures how well the model separates fraud and non-fraud across thresholds.

Prediction threshold:

    if fraud_probability >= threshold:
        prediction = FRAUD
    else:
        prediction = CLEAN

Planned threshold search:

    start = 0.30
    end = 0.70
    step = 0.05

The model will test thresholds like:

    0.30, 0.35, 0.40, ..., 0.70

Then it will choose the threshold that gives the best validation F1 score.


7. Drift simulation formulas
----------------------------

Because the Kaggle dataset is static, we need simulated drift.

Drift means:

    New incoming data starts looking different from the training data.

Configured drift values:

    base_intensity = 0.1
    current_intensity = 0.1
    intensity_increment = 0.2
    max_intensity = 1.0
    drift_threshold = 0.05

Intensity update:

    next_intensity = min(current_intensity + intensity_increment, max_intensity)

Example:

    Week 1: 0.1
    Week 2: 0.3
    Week 3: 0.5
    Week 4: 0.7
    Week 5: 0.9
    Week 6: 1.0

Amount drift:

    drifted_amount = TransactionAmt * (1 + 0.60 * drift_intensity)

Examples:

    intensity = 0.1 means amount increases by about 6%
    intensity = 0.5 means amount increases by about 30%
    intensity = 1.0 means amount increases by about 60%

Velocity drift:

    drifted_velocity_1hr = velocity_1hr * (1 + 2.0 * drift_intensity)

Examples:

    intensity = 0.1 means velocity increases by about 20%
    intensity = 0.5 means velocity increases by about 100%
    intensity = 1.0 means velocity increases by about 200%

Fraud rate drift:

    target_fraud_rate moves from about 3.5% toward about 7% to 8%

One possible formula:

    target_fraud_rate = base_fraud_rate + drift_intensity * (max_fraud_rate - base_fraud_rate)

Example:

    base_fraud_rate = 0.035
    max_fraud_rate = 0.08
    intensity = 0.5

    target_fraud_rate = 0.035 + 0.5 * (0.08 - 0.035)
    target_fraud_rate = 0.0575
    target_fraud_rate = 5.75%

Drift detection rule:

    if p_value < drift_threshold:
        drift_detected = true
    else:
        drift_detected = false

Current threshold:

    drift_threshold = 0.05


8. Retraining formulas and rules
--------------------------------

Promotion rule:

    improvement = new_f1 - old_f1

    if improvement > promotion_min_f1_delta:
        promote new model
    else:
        keep current model

Current value:

    promotion_min_f1_delta = 0.01

This means the new model must improve F1 by more than 1 percentage point.

Retraining safety rules:

    minimum time between retrains = 6 hours
    maximum retrains per day = 3

These rules prevent the system from retraining too often.

    amount_imputed = TransactionAmt if not null else median(TransactionAmt_non_null)
* Identity Left Join:
  - Drops duplicate TransactionID rows from the identity table.
  - Merges the cleaned transaction table with the identity table.
  - Formula:
    joined_df = cleaned_transaction_df.merge(cleaned_identity_df, on='TransactionID', how='left')
* Summary Collection: Generates IngestionSummary metrics including:
  - Raw counts of transaction and identity tables.
  - Cleaning metrics (null transaction IDs count, duplicate transaction IDs count, null amount count).
  - Fraud Rate formula (only on train split):
    fraud_rate = count(isFraud == 1) / total_transaction_rows

Completed Work:

2. Built a batch generation CLI tool in scripts/generate_inference_batch.py.
3. Created a complete test suite in tests/test_drift_simulator.py.

Implemented Logic and Formulas:
* Transaction Amount Drift:
  amount_drifted = TransactionAmt * (1.0 + 0.60 * drift_intensity)
* Velocity Drift:
  velocity_drifted = velocity_1hr * (1.0 + 2.0 * drift_intensity)
* Target Fraud Rate Drift:
  target_fraud_rate = base_fraud_rate + drift_intensity * (max_fraud_rate - base_fraud_rate)
  where base_fraud_rate = 0.035, max_fraud_rate = 0.080.
  Flip clean records (isFraud == 0) to fraud (isFraud == 1) with probability:
  p_flip = (target_fraud_rate - curr_fraud_rate) / (1.0 - curr_fraud_rate)
* Hour Peak Daytime Shift:
  With probability = drift_intensity, fraud transaction hour is shifted toward daytime peak (9:00 to 17:00, centered at 12:00 via a normal distribution).
* Merchant Transaction Count Spike:
  With probability = 0.30 * drift_intensity, ProductCD, card4, and P_emaildomain are set to dominant values (W, visa, gmail.com) to spike rolling merchant volume.


24. Phase 5 Progress: Drift Detection & Automated Retraining

Drift Detection and Auto-Retraining is 100% complete. Implemented statistical drift checks, Evidently HTML reporting, concurrency locks, rate limits, F1 promotion guards, and auto-retraining workflows.

Completed Work:
1. Developed the drift detection module in src/drift_detector.py.
2. Developed the retraining orchestrator in src/retraining_pipeline.py.
3. Built CLI tools scripts/check_drift.py and integrated retraining pipeline triggers.
4. Created a complete test suite in tests/test_drift_detector.py.
5. Verified all statistical tests, concurrency locks, and fallback logging pass successfully.

Implemented Logic and Formulas:
* Numerical Feature Drift:
  Uses the Kolmogorov-Smirnov (KS) test comparing baseline and batch distributions of amount, velocity_1hr, and user_avg_amount_30d.
* Categorical Feature Drift:
  Uses the Chi-Square test of independence on card_type and hour_bucket (bucketed as night [0-6], morning [7-12], afternoon [13-18], evening [19-23]).
* Concurrency Guard:
  Exclusive file creation lock models/retrain.lock prevents parallel retraining runs.
* Retraining Rate Limits:
  - Hourly gap limit: >= 6 hours since last successful retrain.
  - Daily frequency limit: < 3 retrains in the last 24 hours.
* Performance-Based Promotion:
  new_model_f1 - production_model_f1 >= promotion_min_f1_delta (delta = 0.01)
  On successful promotion, current_intensity in config.yaml is reset to base_intensity.
* Logging & Fallbacks:
  Logs drift events to PostgreSQL drift_logs (fallback: logs/drift_logs.json).
  Logs retraining results to PostgreSQL retrain_logs (fallback: logs/retraining_history.json).

Bug 3: Missing target column guard (src/data_preprocessing.py)
   When is_train=True but the isFraud column is absent, the code silently set
   y=None and crashed with a confusing TypeError inside train_test_split.
   Fixed: now raises a clear ValueError with a descriptive message.

Bug 4: NaN labels from inference batch concat (src/retraining_pipeline.py)
   Inference batch rows have no isFraud labels. When concatenated with training
   rows, those rows get isFraud=NaN, causing SMOTE to crash.
   Fixed: added a dropna(subset=['isFraud']) step before preprocess_data().

Bug 5: Evidently column mismatch (src/drift_detector.py)
   The Evidently DataDriftPreset report used ref_card_col for both reference and
   current data. If the inference batch had card_type but the baseline had card6
   (or vice versa), this would raise a KeyError.
   Fixed: each dataset now uses its own resolved column name, and the current
   column is renamed to match the reference name before passing to Evidently.

Bug 6: is_weekend type mismatch (src/api/main.py)
   is_weekend was computed as int (0 or 1) but the ORM column is Boolean.
   Fixed: now computed as bool explicitly.

Bug 7: retrain.yml was a placeholder (.github/workflows/retrain.yml)
   The scheduled GitHub Actions workflow only had an echo statement.
   Fixed: now runs the full pipeline (install deps, generate batch, check drift).
   DATABASE_URL and MLFLOW_TRACKING_URI are read from GitHub repository secrets.

All 34 tests continue to pass after these fixes.
flake8 and black --check also pass with zero errors.


30. How to Verify the Pipeline is Working
-------------------------------------------

This section explains how to test the full pipeline end to end, step by step.
Activate the .venv first:

    .venv\Scripts\activate

STEP 1: Run unit tests.
    python -m pytest tests -v
    Expected: 34 tests pass, all green.

STEP 2: Run data ingestion smoke test (no files saved).
    python -m src.data_ingestion
    Expected: prints IngestionSummary for train and test splits.
              Train fraud_rate should be about 0.035.

STEP 3: Run feature engineering and preprocessing.
    WARNING: 5-10 minutes on the full 590k row dataset.

    python scripts/run_preprocessing.py
    (or run manually via python -c as a one-liner)
    Saves: data/processed/ CSVs and models/ transformer files.

STEP 4: Train the model.
    python scripts/train.py
    Expected: prints trial F1s and saves models/xgboost_model.joblib.
    MLflow falls back to ./mlruns/ if Docker is not running.

STEP 5: Evaluate the model.
    python scripts/evaluate.py
    Expected: prints metrics, saves reports/evaluation_report.json.

STEP 6: Generate a drifted inference batch.
    python scripts/generate_inference_batch.py
    Expected: saves data/inference_batches/batch_YYYYMMDD.csv.

STEP 7: Run drift detection.
    python scripts/check_drift.py
    Expected: prints KS and Chi-Square test results.
              Triggers retraining automatically if any p-value < 0.05.

STEP 8: Start FastAPI locally.
    python -m uvicorn src.api.main:app --host 0.0.0.0 --port 8000 --reload
    Then open: http://localhost:8000/docs
    Health:    http://localhost:8000/health

STEP 9: Start full Docker stack (requires Docker Desktop running).
    docker compose up --build
    Services:
        PostgreSQL:  localhost:5432
        MLflow UI:   http://localhost:5001
        FastAPI API: http://localhost:8000
        Grafana:     http://localhost:3001  (admin / admin)

SHORTCUT: Use the Makefile.
    make setup     Install dependencies
    make up        Start full Docker stack
    make train     Run training pipeline
    make evaluate  Run evaluation
    make drift     Generate batch and check drift
    make retrain   Manually trigger retraining
    make test      Run tests
    make lint      Run lint checks
    make api       Start FastAPI locally
    make clean     Remove generated files and caches
